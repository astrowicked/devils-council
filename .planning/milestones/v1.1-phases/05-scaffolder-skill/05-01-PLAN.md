**Frontmatter (exact values):**
The YAML frontmatter block must contain these exact fields:
  name: create-persona
  description: "Interactive scaffolder for custom devils-council personas. Collects voice-kit fields, validates schema, coaches voice distinctness. Writes to plugin data workspace."
  allowed-tools: [AskUserQuestion, Read, Write, Bash, Glob]
  user-invocable: true
  argument-hint: "[persona-slug]"

**Body structure -- the following sections in this order:**

1. **Intro paragraph:** Brief explanation of what the skill does. Reference `agents/AUTHORING.md` as the manual equivalent. State that this wizard automates the authoring guide.

2. **Step 0: Slug and Workspace Check (D-07)**
   - If `$ARGUMENTS` is provided, use as the slug. Otherwise, use AskUserQuestion to collect persona name.
   - Validate slug is kebab-case: `[a-z0-9]+(-[a-z0-9]+)*` (no uppercase, no underscores, no spaces, no `../`).
   - Check if `${CLAUDE_PLUGIN_DATA}/create-persona-workspace/<slug>/` already exists via Bash tool `test -d`.
   - If exists, use AskUserQuestion: "A persona workspace for '<slug>' already exists. Overwrite?" with options: "Overwrite (start fresh)" / "Cancel".
   - If cancel, stop.

3. **Step 1: Tier Selection (per D-01 linear wizard)**
   - Use AskUserQuestion with Header: "Tier", Question: "Which tier is this persona?"
   - Options: "core -- Always invoked on every review" / "bench -- Auto-triggered by structural signals"
   - Do NOT offer chair or classifier tiers (those are internal).
   - If bench, proceed to Step 1b: Triggers.

4. **Step 1b: Triggers (bench tier only) (per D-03)**
   - Read signal IDs from `lib/signals.json` via Bash tool: `jq -r '.signals | to_entries[] | "\(.key) -- \(.value.description)"' "${CLAUDE_PLUGIN_ROOT}/lib/signals.json"`
   - Present the full list to the user. Use AskUserQuestion: "Which signal(s) trigger this persona? List one or more signal IDs separated by commas."
   - Validate each ID exists in signals.json. If invalid, re-ask.

5. **Step 2: Primary Concern (per D-01, D-03)**
   - Use AskUserQuestion: "What is this persona's primary concern? (One sentence ending with '?' -- the value-system anchor)"
   - Show existing persona primary_concerns as negative examples (what NOT to duplicate) -- read from persona-metadata/*.yml via Bash tool: `for f in ${CLAUDE_PLUGIN_ROOT}/persona-metadata/*.yml; do yq -r '.primary_concern // empty' "$f" 2>/dev/null || python3 -c "import yaml,sys; d=yaml.safe_load(open(sys.argv[1])); print(d.get('primary_concern',''))" "$f"; done`
   - Validate: must end with `?`, must be non-empty.

6. **Step 3: Blind Spots**
   - Use AskUserQuestion: "List 2-4 things this persona explicitly does NOT care about (blind spots). One per line."
   - Must be a non-empty list.

7. **Step 4: Characteristic Objections (per D-01, D-05 inline coaching)**
   - Use AskUserQuestion: "List at least 3 verbatim phrases this persona would actually say as objections. These are exact strings, not paraphrases. One per line."
   - Enforce minimum count >= 3. If fewer, explain and re-ask.
   - **Inline coaching (D-05):** After collecting, check each objection against the banned_phrases collected in the NEXT step. Since banned_phrases comes after in the wizard, do this check AFTER Step 5 collects banned phrases. If any objection contains a banned phrase (case-insensitive substring), flag it: "Your objection '[objection]' uses a phrase you banned ('[phrase]'). Rephrase or keep as-is?" Use AskUserQuestion with options: "Rephrase (ask me for a replacement)" / "Keep (I intentionally quote it)".

8. **Step 5: Banned Phrases (per D-01, D-03, D-04 overlap coaching)**
   - Use AskUserQuestion: "List at least 5 phrases this persona should NEVER use in findings. Start with the three baseline bans: 'consider', 'think about', 'be aware of', then add role-specific bans. One per line."
   - Show baseline as starting point per D-03.
   - Enforce minimum count >= 5 (per ROADMAP SC-1, stricter than validator's R6 >= 1). If fewer, explain and re-ask.
   - **Overlap coaching (D-04):** After collecting, compute overlap with all 16 shipped persona sidecars. Use Bash tool with python3:
     `python3 -c "import yaml, sys, os, glob; user_bans = set(b.strip().lower() for b in sys.argv[1].split('|') if b.strip()); baseline = {'consider', 'think about', 'be aware of'}; user_role_specific = user_bans - baseline; metadata_dir = sys.argv[2]; [print(f'{os.path.basename(f).replace(\".yml\",\"\")}: {len((user_role_specific & (set(b.lower() for b in (yaml.safe_load(open(f)) or {}).get(\"banned_phrases\",[])) - baseline)))*100//min(len(user_role_specific),len(set(b.lower() for b in (yaml.safe_load(open(f)) or {}).get(\"banned_phrases\",[])) - baseline))}% overlap') for f in sorted(glob.glob(os.path.join(metadata_dir, '*.yml'))) if (lambda shipped_role: shipped_role and user_role_specific and len(user_role_specific & shipped_role)*100//min(len(user_role_specific),len(shipped_role)) > 30)(set(b.lower() for b in (yaml.safe_load(open(f)) or {}).get('banned_phrases',[])) - baseline)]" "PIPE_SEPARATED_BANS" "${CLAUDE_PLUGIN_ROOT}/persona-metadata"`
   - However, for READABILITY in the SKILL.md body, write the overlap detection as a multi-line python3 script passed to Bash tool (NOT as a one-liner). The SKILL.md body instructs Claude to invoke Bash with a readable python3 script that: reads all persona-metadata/*.yml, computes set-intersection of role-specific banned phrases (after removing baseline set of consider/think about/be aware of), reports any persona with >30% overlap along with the specific overlapping phrases.
   - If overlap detected, show which phrases overlap with which persona, explain why differentiation matters, use AskUserQuestion: "Your banned-phrase set has >30% overlap with [persona]. Keep these or diversify?" Options: "Keep as-is" / "Diversify (ask me for replacements)".

   - **Then run D-05 check:** Now that both objections and banned phrases are collected, cross-check per Step 4 coaching logic.

9. **Step 6: Worked Examples (per D-01)**
   - Use AskUserQuestion: "Provide 2 good-finding examples (what this persona SHOULD produce). Each needs: target (specific line/heading), claim (in persona voice), evidence (verbatim 8+ char substring), ask (actionable remediation), severity, category."
   - Then: "Provide 1 bad-finding example (what this persona REFUSES to ship). Same fields, plus a rejection reason."
   - Enforce: exactly 2 good + 1 bad minimum. If fewer, explain and re-ask.

10. **Step 7: End Preview (D-02)**
    - Assemble the full agent file content (all-in-one legacy format with ALL custom fields in frontmatter for validation).
    - Show the complete file to the user.
    - Use AskUserQuestion: "Here is the complete persona file. Ready to write?" Options: "Write it" / "Edit (go back to a specific step)" / "Cancel".
    - If edit, ask which step to revisit and loop back.

11. **Step 8: Write to Workspace (D-06, D-07)**
    - Create workspace directory: `${CLAUDE_PLUGIN_DATA}/create-persona-workspace/<slug>/agents/` and `.../persona-metadata/`
    - Write the all-in-one agent file to `<workspace>/agents/<slug>.md` (legacy format -- all custom fields in frontmatter for validation).
    - No sidecar yet -- validation uses legacy fallback.

12. **Step 9: Validate (D-08, D-09, SCAF-03)**
    - Run validator via Bash tool: `"${CLAUDE_PLUGIN_ROOT}/scripts/validate-personas.sh" "${workspace}/agents/${slug}.md" --signals "${CLAUDE_PLUGIN_ROOT}/lib/signals.json"`
    - If exit 0: proceed to split step.
    - If exit 1: parse error output, map each R-code to the field question per this table:
      R1 = "YAML frontmatter is malformed" = go back to preview
      R2 = "Invalid tier value" = go back to Step 1
      R3 = "primary_concern is empty" = go back to Step 2
      R4 = "blind_spots is empty or missing" = go back to Step 3
      R5 = "Need at least 3 characteristic_objections" = go back to Step 4
      R6 = "banned_phrases is empty" = go back to Step 5
      R7 = "Trigger ID not found in signals.json" = go back to Step 1b
      R8 = "Core/chair personas must not have triggers" = go back to Step 1
    - Allow up to 3 retry loops. After 3 failures, write anyway with a warning: "Persona written but did not pass validation. Fix manually using: ./scripts/validate-personas.sh <workspace>/agents/<slug>.md"

13. **Step 10: Split into Agent + Sidecar**
    - After validation passes, split the all-in-one file:
      - `<workspace>/agents/<slug>.md` retains only: name, description, tools (default: [Read, Grep, Glob]), model (inherit), skills ([persona-voice, scorecard-schema]), plus the markdown body.
      - `<workspace>/persona-metadata/<slug>.yml` gets: tier, primary_concern, blind_spots, characteristic_objections, banned_phrases, tone_tags, triggers (if bench).
    - Write both files via the Write tool.

14. **Step 11: Print Install Commands (D-06)**
    - Print exact commands:
      `cp "${CLAUDE_PLUGIN_DATA}/create-persona-workspace/<slug>/agents/<slug>.md" <your-plugin-dir>/agents/`
      `cp "${CLAUDE_PLUGIN_DATA}/create-persona-workspace/<slug>/persona-metadata/<slug>.yml" <your-plugin-dir>/persona-metadata/`
    - Remind user: "After copying, run `./scripts/validate-personas.sh` to verify, then `/reload-plugins` to pick up the new persona."
